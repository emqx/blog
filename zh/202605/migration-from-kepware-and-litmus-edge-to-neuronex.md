## 迁移至 NeuronEX 的技术考量

从传统连接层网关或边缘一体机向现代边缘数据平台演进时，典型约束可按维度归纳如下：

| **维度**           | **常见结构性约束**                                           |
| :----------------- | :----------------------------------------------------------- |
| **部署与交付**     | 与特定操作系统生态绑定导致硬件选型收窄；缺乏容器化与编排支持时，多站点一致性交付难度大 ； |
| **架构与组件边界** | 套件中与数据采集弱相关的组件叠加，推高资源占用与运维复杂度 ； |
| **采集与流处理**   | 超高频、海量点位场景下，采集链路效率与边缘流处理稳定性易成为瓶颈 ； |

**NeuronEX** 的对应能力取向如下：

| **能力域**     | **要点**                                                     |
| :------------- | :----------------------------------------------------------- |
| **运行与编排** | Linux 原生；支持 x86 / ARM；以 Docker / Kubernetes 交付；支持镜像级复制与标准化运维；有助于控制与操作系统授权栈相关的总拥有成本（TCO） |
| **南向采集**   | 工业边缘数据采集网关定位；集成百余类南向协议驱动；在高并发、大规模 Tag 场景下平衡采集吞吐与资源占用 |
| **边缘计算**   | 内置基于 SQL 的流处理引擎；在边缘侧完成数据清洗、聚合、告警及部分推理前置；抑制无效上行、缩短闭环时延 |
| **北向与生态** | 可与 EMQX 无缝协同，基于统一命名空间（UNS）实现云边数据贯通；同时兼容 Azure IoT Edge、AWS IoT Greengrass 等主流云边运行时环境。 |

相较于以连接与协议转换为主的数采方案，NeuronEX 更贴合 **云原生边缘部署** 与 **数据源侧预处理**。

## **驱动配置迁移工具**使用指南

### 导出—转换—导入流程

配置迁移工具将 **KEPServerEX**、**Litmus Edge** 等来源中的驱动与点位配置批量映射为 **NeuronEX 南向可用的 JSON**。建议按以下顺序执行：

1. **盘点**：依据[迁移指南](https://docs.emqx.com/zh/neuronex/latest/configuration/driver-migration-tool.html)核对现有协议是否落入迁移支持列表，识别已知限制（字符串长度、数组、证书等）。  
2. **导出**：在源平台按文档步骤导出配置文件（Kepware：JSON；Litmus：`Plain Text`）。  
3. **转换**：访问官网配置迁移工具页面，选择正确来源并上传，服务端完成映射与校验后下载结果文件，审阅成功 / 失败汇总。  
4. **导入与联调**：在 NeuronEX Dashboard **南向设备**相关界面导入 JSON，处理失败项或手工补点后，完成连通性与数据质量验证。

迁移完成后，须确认 **NeuronEX 运行环境对目标现场设备的网络可达性**（IP、端口、串口设备及安全策略以 NeuronEX 侧最终配置为准），并按文档在 Dashboard 中完成 **连接状态检查** 与 **数据监控**验证。

### 在线托管形态

配置迁移工具以 **网站托管服务** 形式提供，无需单独安装本地迁移客户端；在网络与安全策略允许的前提下，可用于试点验证（POC）。
